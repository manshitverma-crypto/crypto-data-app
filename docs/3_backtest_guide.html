3_backtest.py

Complete guide to the Backtest page of the Crypto Data App — a Streamlit-based crypto backtesting dashboard built on NautilusTrader.

Streamlit Page NautilusTrader 382 lines 4 Built-in Strategies

This file is Page 3 of a multi-page Streamlit application. It allows users to:

  • Select an instrument/bar type from a pre-loaded data catalog
  • Upload custom trading strategies or choose from 4 built-in ones
  • Configure per-strategy parameters (periods, thresholds, trade size)
  • Run backtests using the NautilusTrader engine
  • View comparison tables, detailed reports, and download HTML/CSV reports

# Architecture

File Dependencies

3_backtest.py orchestrates several core modules:

pages/
  3_backtest.py ← This file (UI + orchestration)

core/
  nautilus_loader.py ← load_catalog() — loads ParquetDataCatalog
  custom_strategy_loader.py ← validate, load, merge custom strategies
  strategies.py ← 4 built-in strategies + STRATEGY_REGISTRY
  backtest_runner.py ← run_backtest() — engine wrapper
  report_generator.py ← generate_report() — HTML report

docs/
  report_template.html ← Chart.js template for HTML reports

custom_strategies/ ← User-uploaded .py strategy files
catalog/ ← ParquetDataCatalog (bars + instruments)
reports/ ← Auto-saved HTML + CSV reports

Data Flow

1. Load Catalog

Read ParquetDataCatalog from disk → extract available bar types (e.g. BTCUSD.YAHOO-1-DAY-LAST-EXTERNAL)

2. Build Strategy Registry

Merge 4 built-in strategies + any validated custom uploads → unified registry dict

3. User Configures

Select bar type, starting capital, strategies, and per-strategy parameters via Streamlit UI

4. Run Backtests

For each selected strategy: configure NautilusTrader BacktestEngine → run → extract results

5. Generate Reports

Build HTML report (Chart.js) + auto-save CSV files (positions, fills, account) per strategy

6. Display Results

Show comparison table, download button, per-strategy metrics, positions/fills/account DataFrames

Step 5: Upload Validation Pipeline Lines 85–131

When a file is uploaded, a multi-phase validation process runs:

Sanitize Filename

sanitize_filename() strips unsafe characters → keeps only [a-zA-Z0-9_\-.], collapses multiple underscores

Save to Disk

File bytes are written to custom_strategies/{safe_name}.py

Validate & Load

validate_and_load_strategy() runs 6-phase validation (see Custom Strategy Loader section)

Duplicate Name Check

Iterates all other .py files in custom_strategies/ to check if any share the same STRATEGY_NAME

Success or Cleanup

On success: show green message. On failure: delete the file and show error. Prevents invalid strategies from persisting.

File Replacement
If a file with the same name already exists, it is overwritten and a warning is shown: "Replaced existing file: {name}".

Execution Flow
  1. run_backtest is imported lazily (only when the button is clicked) to avoid loading NautilusTrader on every page render
  2. Strategies run sequentially with a progress bar showing current strategy index
  3. Each call to run_backtest() creates a fresh BacktestEngine, runs it, and returns a results dict
  4. Errors are caught per-strategy — one failure doesn't stop other strategies from running
  5. Results are stored in all_results dict keyed by strategy name

# Built-in Strategies

All strategies are defined in core/strategies.py and follow the same pattern: a frozen StrategyConfig subclass + a Strategy subclass with on_start, on_bar, on_stop methods.

EMA Cross

Buy when fast EMA crosses above slow EMA; sell when it crosses below. Classic trend-following strategy.

Params: fast_ema_period (2–100, default 10), slow_ema_period (5–200, default 20) 
# Trading logic (on_bar):
if fast_ema.value >= slow_ema.value:
    # BUY signal (close short if any, open long)
elif fast_ema.value < slow_ema.value:
    # SELL signal (close long if any, open short)

RSI Mean Reversion

Buy when RSI drops to oversold levels; sell when RSI reaches overbought. Contrarian reversal strategy.

Params: rsi_period (2–50, default 14), overbought (50–95, default 70), oversold (5–50, default 30) 
# Trading logic (on_bar):
if rsi.value <= config.oversold:
    # BUY signal
elif rsi.value >= config.overbought:
    # SELL signal

Bollinger Bands

Buy at lower band; sell at upper band. Band-based mean reversion assuming price stays within bands.

Params: bb_period (5–100, default 20), bb_std (0.5–4.0, default 2.0) 
# Trading logic (on_bar):
if close <= bb.lower:
    # BUY signal
elif close >= bb.upper:
    # SELL signal

4 Moving Averages

Buy when all 4 MAs align bullish (MA1 > MA2 > MA3 > MA4); sell on bearish alignment. Supports SMA/EMA toggle.

Params: use_ema (bool), ma1 (5), ma2 (10), ma3 (20), ma4 (50) 
# Trading logic (on_bar):
if v1 > v2 > v3 > v4:
    # BUY signal (full bullish alignment)
elif v1 < v2 < v3 < v4:
    # SELL signal (full bearish alignment)
Common Strategy Pattern
All strategies share identical position management logic:
  • If flat (no position): open a new position in the signal direction
  • If opposite position exists: close it first, then open the new one
  • If same direction: do nothing (already in the right position)
  • On stop: cancel all orders + close all positions

Result Extraction

The _extract_results() function uses a 3-tier fallback system for counting wins/losses:

Tier 1: Positions Report

Parse realized_pnl column from generate_positions_report(). Handles NautilusTrader's "150.00 USD" string format by splitting on whitespace.

Tier 2: Cache Positions

If positions report was empty, iterate closed positions from engine.kernel.cache.positions() and read pos.realized_pnl.

Tier 3: Fills Estimate

Last resort: estimate trades as total_orders // 2 and assign single win or loss based on total P&L sign.

# Custom Strategy Loader (custom_strategy_loader.py)

6-Phase Validation Pipeline

Phase 1: File Checks

File exists? Not empty?

Phase 2: Module Loading

Dynamically import with importlib.util. Catches SyntaxError, ImportError.

Phase 3: Required Exports

Must have: STRATEGY_NAME, STRATEGY_CLASS, CONFIG_CLASS, DESCRIPTION, PARAMS

Phase 4: Type Checks

STRATEGY_NAME: str (max 100 chars), CONFIG_CLASS: subclass of StrategyConfig, STRATEGY_CLASS: subclass of Strategy

Phase 5: Config & Method Validation

Config must have instrument_id, bar_type, trade_size fields. Strategy must define on_start(), on_bar(), on_stop().

Phase 6: PARAMS Validation

Each param needs label + default. Numeric params need min/max. Defaults must be in range. Keys must match CONFIG_CLASS fields.

Error Collection
Phases 3–6 collect all errors before raising a single ValueError. This means the user sees every problem at once instead of fixing them one at a time.

Strategy Template

get_strategy_template() returns a complete, well-commented Python file (~160 lines) that includes:

  • A sample MyStrategyConfig class with required + custom fields
  • A sample MyStrategy class with all required methods implemented
  • All 5 required exports at the bottom
  • A complete PARAMS dict demonstrating int, float, and bool parameter types
  • Inline comments explaining every section
